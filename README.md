# DeepSeek 互动小说工作台

这是一个基于 Gradio 的互动小说写作工具。用户作为“故事导演”给出评价、修正和设定调整，AI 负责生成剧情草稿；只有用户点击“接受并保存”后，内容才会进入正式故事记录。

项目当前只维护 Web UI 流程，不提供独立 CLI 写作流程。

## 主要功能

- 构建并保存世界观与角色卡。
- 按故事目录管理多个剧本，所有故事放在 `Stories/` 下。
- 故事模式中生成剧情草稿、修订草稿、接受保存或删除上一个已通过回复。
- 支持继续已有场景，直接读取 `scene_NN.txt` 回到故事模式。
- 自动维护 `outline.txt` 和当前场景 summary，帮助长剧情保持连续性。
- 支持 DeepSeek 模型、思考模式和思考强度设置。
- 支持特殊命令 `/k`、`/s`、`/u`、`/c`、`/w`。

## 启动

建议使用 Python 3.9+。

```bash
python3 run.py
```

启动后浏览器会打开本地 Gradio 页面。默认监听本机地址，端口会自动选择可用端口，通常是：

```text
http://127.0.0.1:7860
```

## API 设置

在「启动 / API」页面填写 DeepSeek API Key。

可选模型：

- `deepseek-v4-pro`，默认模型。
- `deepseek-v4-flash`。

可选思考设置：

- 思考模式：`enabled` / `disabled`。
- 思考强度：`high` / `max`。

故事进行中也可以切换 API 页面修改模型和思考设置，不会重置当前剧情状态。

## 故事目录

所有剧本放在：

```text
Stories/
```

每个剧本是一个独立文件夹，例如：

```text
Stories/Story1/
```

常用文件结构：

```text
Stories/Story1/
├── World.txt          # 世界观
├── outline.txt        # 已完成章节/场景提纲
├── scene_01.txt       # 场景正式记录
├── scene_02.txt
└── char/
    ├── char1.txt      # 角色卡
    └── char2.txt
```

程序只改代码逻辑，不负责替用户迁移旧故事文件。

## 基本使用流程

1. 在「启动 / API」中选择故事目录并填写 API Key。
2. 在「世界 / 角色」相关页面创建或编辑 `World.txt` 和角色卡。
3. 进入「故事模式」。
4. 可以填写场景简描后快速开始，也可以留空无大纲开始。
5. 用户输入导演评价或指令，AI 生成草稿。
6. 如果满意，点击「接受并保存」。
7. 如果不满意，直接继续输入修改意见，AI 会基于前一版草稿继续修订。
8. 结束本幕时点击「结束本幕」进行结算和 outline 更新。

## 草稿与正式记录

故事模式有明确的临时/正式边界：

- AI 回复后只是草稿。
- 点击「接受并保存」后，草稿才进入正式记录。
- 未接受时继续输入，属于同一个临时修订链。
- 临时修订链最终只保存最后通过的 AI 正文。
- 被拒绝的草稿和用户修改意见不会写入 `scene_NN.txt`。

`scene_NN.txt` 只保存正式 AI 正文，例如：

```text
**EVENT:**
...

**CORRECTION:**
...

**SCENE SYNOPSIS:**
...
```

不会保存导演输入、废弃草稿或 UI 样式 HTML。

## 特殊命令

特殊命令必须出现在输入开头才会生效。

| 命令 | 作用 |
|------|------|
| `/k` | 接受当前草稿并让 AI 继续自由推演下一段。 |
| `/s [说明]` | 生成当前场景后续大纲。若已有未接受草稿，则作为同一临时修订链的一部分，不会自动保存上一版。 |
| `/u <修正>` | 上帝指令，用最高优先级修正当前内容。若已有未接受草稿，则只作为当前草稿的强修订意见；没有未接受草稿时，接受后保存为 `CORRECTION`。 |
| `/c中文文本` | 在故事中临时创建或更新角色卡，确认后写入角色文件，不进入剧情记录。 |
| `/w中文文本` | 在故事中临时更新世界设定，确认后写入 `World.txt`，不进入剧情记录。 |

`普通文字/c修改` 不会触发 `/c`，因为命令不在开头。

## 上下文与长场景

为了减少模型混乱和 token 浪费，故事写作上下文采用轻量结构：

- 主提示。
- 世界观。
- 登场人物卡。
- 当前场景。
- `outline.txt`。
- 当前场景 summary。
- 最近正式剧情历史。
- 当前临时修订链。

最近正式历史按字符数控制，不按段落数或消息条数控制。超过阈值后，程序会让 AI 生成当前场景 summary；随后本地按换行段落边界压缩 `writing_history`，只保留最近现场细节。

裁剪只影响 API 上下文，不会删除 `scene_NN.txt` 中的正式记录。

## UI 标记

故事模式聊天框中：

- 浅琥珀背景：导演输入或未通过草稿。
- 浅绿色背景：已通过的正式 AI 正文。

这些颜色只用于 UI 显示，不会写入故事文件。

## 删除与继续

- 「删除上一个回复」会回滚最近一次已通过 AI 回复，同时回滚正式记录、上下文 summary 和相关状态。
- 「继续已有场景」会从 `scene_NN.txt` 读取正式记录，恢复到故事模式并跳过导演会议阶段。

## 维护说明

- API Key 只保存在运行时内存中，不写入磁盘。
- `webui.py` 是 Gradio UI 入口，应保持轻量。
- 核心逻辑按模块放在 `story_core/` 中。
- 修改后建议运行：

```bash
PYTHONPYCACHEPREFIX=/private/tmp/deepseekNovel_pycache python3 -m py_compile config.py webui.py run.py story_core/*.py
```
